# 게시글 생성 폼

게시글이 해야하는 일, 모델에 맞춰서
- 제목 입력
- 내용 입력
- Userid ( ) 로그인 사용자 기준 처리
- 저장(Store)  / 저장 성공시 post index로 이동

 validated로 검증, required = 필수 / max:255 = 255자 이하

fillable쪽에서의 허용 / validate에서의 검증 양측?

spring이었으면 validate는 DTO와 같은 역할? 데이터 사전 값 정의

fillable DTO -> Entity 들어가는 값 조절

##  Laravel 흐름
1. 요청값 받음

2. 요청값 검증

3. 저장 가능한 값 선별

4. Entity(Model)에 넣음

5. DB 저장


## 스프링 흐름

1. 요청값 /posts로 받음
2. request값 검증 @Valid DTO
3. 저장 가능한 객체면 request 아니면 에러처리 및 리다이렉트
4. 새 Entity 만듦(db 매핑할 모델객체)
5. 해당 엔티티에 받아온 값 매핑
6. repository.save()로 DB 저장



| 개념 | Laravel | Spring MVC |
|---|---|---|
| 요청 데이터 받기 | `Request $request` | DTO 파라미터 |
| 입력 검증 | `$request->validate()` | `@Valid` + Bean Validation |
| 검증 통과 데이터 | `$validated` 배열 | DTO 객체 |
| 현재 로그인 사용자 | `Auth::id()` | `@AuthenticationPrincipal`, `SecurityContext` |
| 엔티티 저장 허용 필드 | `$fillable` | DTO → Entity 매핑 코드로 통제 |
| 엔티티 생성/저장 | `Post::create()` | `new Entity()` + `repository.save()` |

# 게시글 수정 폼

목표는 Update

## 읽어와야 하는것 + 수정 가능한 것
    -   기존 제목
    -   기존 내용
## 제공하는 기능
    -   수정(Update) -> 이후 라우팅?(목록 or 해당 글 상세보기 폼)

## 수정 요청 흐름
    1.  수정페이지 call
    2.  기존게시글 READ
    3.  기존 값 출력
    4.  수정 진행(완료시)
    5.  수정 요청 call
    6.  요청 검증(id 일치) -> 허용된것만 반영
    7.  DB Update
    8.  성공시 페이지 라우팅 / 실패시 abort?
